Codex Skill の対話モード（`codex exec` + `codex exec resume`）を使って、設計に対する対立的レビュー（Adversarial Review）を実施してください。

Codex には**賛同者ではなく批判者**として振る舞わせる。設計の妥当性を確認するのではなく、設計が破綻する条件を積極的に探させる。

## 手順

### 1. 文脈の準備

- ユーザーが指定したトピックに関連するファイルを特定する
- 設計意図・検討した選択肢・採用理由を整理する

### 2. 対立的レビューの開始

`codex exec --full-auto --cd <dir> --json` でセッションを開始する。以下を渡す:

- 対象ファイルの現在の内容
- 設計意図と採用した選択肢
- **以下の対立的な問いへの回答を要求する**:

```
あなたはこの設計の批判的レビュアーです。賛同ではなく、設計の弱点を突くことが役割です。

1. **破綻シナリオ**: この設計が失敗する具体的なシナリオを3つ挙げてください。
   各シナリオについて、発生条件・影響範囲・発生確率（高/中/低）を示してください。

2. **隠れた前提**: この設計が暗黙に依存している前提を列挙してください。
   その前提が崩れた場合に何が起きるかも述べてください。

3. **対案の提示**: この設計とは根本的に異なるアプローチを1つ以上提案してください。
   現案と対案を以下の軸で比較してください:
   - 実装コスト
   - 保守性
   - 拡張性
   - 障害時の影響範囲

4. **見送った選択肢の検証**: 設計者が検討して却下した選択肢がある場合、
   その却下理由に反論してください。却下は本当に正しかったですか？

5. **最悪ケース**: この設計を採用した場合に起こりうる最悪の結末を描写してください。
   それを防ぐために必要なガードレールは何ですか？
```

### 3. 深掘り対話

`codex exec resume --last --full-auto --json` で以下を追加で問う:

- ステップ2の回答で最も深刻だと判断したリスクについて、具体的な緩和策を提案させる
- 対案が現案より優れている点を、コード例を交えて説明させる
- 「それでも現案を採用すべき理由」があるなら述べさせる（公平性の担保）

### 4. 報告

結論を以下の構造でユーザーに報告する:

| 項目 | 内容 |
|------|------|
| **判定** | 続行 / 要修正 / 再設計推奨 |
| **最大リスク** | 最も深刻な破綻シナリオとその緩和策 |
| **隠れた前提** | 見落としていた暗黙の依存 |
| **対案の価値** | 対案に採り入れるべき要素があるか |
| **推奨アクション** | 設計を進める前にやるべきこと |
